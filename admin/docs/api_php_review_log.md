# Журнал проверки PHP API для Native Places

Дата: 2026-06-27

Этот файл собран как рабочая памятка по PHP endpoints, которые были просмотрены и для которых были предложены исправленные версии. Цель — быстро перенести решения в `docs` основного проекта и не потерять договорённости по полям, статусам и логике.

## Общие правила, которые применяли ко всем PHP endpoints

- Проверять, что `json_decode(..., true)` вернул массив, иначе отвечать `Некорректный JSON` с HTTP 400.
- Использовать единый формат ответа через `successResponse()` и `errorResponse()`.
- Не отдавать debug JSON вручную из catch-блоков; использовать общий `errorResponse()`.
- В write/update/delete endpoints применять транзакции, если вместе с изменением пишется лог или несколько связанных записей.
- Не строить SQL из пользовательских значений, кроме заранее проверенных технических чисел вроде `LIMIT`.
- Все внешние фильтры (`status`, `category`, `type`, `booking`, `code`) валидировать whitelist/regex.
- Не полагаться только на фронт: backend должен сам проверять роль, статус пользователя, существование сущностей и связи между ними.
- Для действий модераторов/админов писать запись в `moderator_logs`, где это влияет на данные.

## Auth и shared

### `/api/admin/auth/login-admin.php`

Что исправляли/рекомендовали:

- Проверять корректность JSON.
- Проверять email/password.
- Проверять `users.status = active`.
- Проверять роль `admin` через таблицу `roles`.
- После успешного входа делать `session_regenerate_id(true)`.
- В сессию класть `access_type = account`.

### `/api/admin/auth/login-code.php`

Что исправляли/рекомендовали:

- Код доступа проверять через `password_verify()` по активным кодам.
- Проверять `expires_at`.
- Проверять, что код привязан к пользователю.
- Проверять, что пользователь существует, активен и имеет роль `moderator`.
- Обновлять `admin_access_codes.last_login_at`.
- После успешного входа делать `session_regenerate_id(true)`.
- В сессию класть `access_type = code` и `access_code_id`.

### `/api/admin/auth/logout.php`

Что исправляли/рекомендовали:

- Удалять `$_SESSION['admin_user']`.
- При необходимости регенерировать session id после выхода.

### `/api/admin/auth/me.php`

Что исправляли/рекомендовали:

- Возвращать `authenticated = false`, если нет `admin_user` в сессии.
- Возвращать текущего пользователя из сессии без повторной авторизации.

### `/api/admin/shared/require-admin.php`

Что исправляли/рекомендовали:

- `requireAdmin()` пропускает только `role_code = admin`.
- `requireAdminOrModerator()` пропускает `admin` и `moderator`.
- В перспективе можно добавить повторную проверку пользователя в БД, но текущая сессионная модель подходит для текущего этапа.

### `/api/admin/shared/moderator-log.php`

Что исправляли/рекомендовали:

- Сделать `writeModeratorLog(..., ?PDO $pdo = null)`, чтобы можно было писать лог в той же транзакции.
- Валидировать `moderator_id` и `entity_id`.
- Ограничивать `action_type` и `entity_type` до 50 символов под структуру таблицы.
- Использовать переданный `$pdo`, если он есть, иначе создавать новое подключение.

### Решение по платежам

- Для текущего этапа можно проектировать систему под онлайн-оплату.
- Для самозанятого предпочтительно рассмотреть ЮKassa для самозанятых или доступный продукт Сбера.
- Логика должна быть: `payments.pending` → успешный webhook → `payments.paid` → создание/активация `user_subscriptions`.
- Квитанции можно не делать, если сразу подключать онлайн-оплату.

## Бизнес-логика тарифов и объявлений

Решение по срокам:

- `plans.max_places` — лимит объявлений в тарифе.
- `plans.duration_days` — срок действия тарифа.
- `user_subscriptions.expires_at` — дата окончания подписки пользователя.
- `places.expires_at` — дата окончания публикации конкретного объявления, обычно равная `user_subscriptions.expires_at`.

Рекомендуемая логика:

1. Пользователь покупает/получает тариф.
2. Создаётся активная `user_subscriptions`.
3. При создании объявления проверяется лимит `plans.max_places`.
4. Если лимит не превышен, объявление создаётся со статусом `pending`.
5. При публикации `places.expires_at` ставится равным `user_subscriptions.expires_at`.
6. Если подписка бессрочная, `user_subscriptions.expires_at = NULL` и `places.expires_at = NULL`.
